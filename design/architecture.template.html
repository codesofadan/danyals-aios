AIOS - SEO Automation PlatformSection 01 - Executive Summary
Section 01
Executive Summary
AIOS turns the agency's SEO delivery into a self-running, cloud-hosted platform across four modules - Audit, Content, the Portal, and the mandatory Policy Radar intelligence engine - with Google Sheets reporting and a Command Center where recommendations surface. AIOS is a product deployed per agency client (Danyal, Haseeb, and beyond); the core is shared. The goal is roughly 90% automation, cutting per-page effort from hours to minutes, with a human review checkpoint on the final step.
Locked decisions - v1
  • Audit module reuses the existing, proven audit engine, wrapped as a cloud job / API.
  • Policy Radar is mandatory core in every deployment - auto-researches Google, flags the knowledge base, feeds the Command Center.
  • Stack: Next.js portal + FastAPI (Python) + Postgres + Redis, on Docker Compose.
  • API keys are pre-loaded by the agency into a central vault; no payment gateway in v1.
  • Publishing: WordPress REST API in v1, plus a manual PDF path. Images AI-generated with alt text.
  • Onboarding: agency-provisioned accounts; no public self-signup.
  • Financial audit report is documented now, built in Phase 2.
  • Off-Page module is out of scope for v1; documented for later.
  • Free vs paid audit tiers are set by the agency and gate the costly integrations.
  • Deliverable is this internal founder-grade architecture + build plan.

The problem

Manual audits and content take hours per page and every automated run burns API budget that is hard to predict.

The system

One cloud platform: click to audit, generate + publish content, and give every client a live portal - all on cost-controlled infrastructure.

The payoff

Minutes per page, a scalable client base, and a branded self-serve experience that funnels upsells to the agency's Fiverr gigs.

System Architecture
A browser talks to the Next.js app. Next.js calls FastAPI for all logic and auth. FastAPI uses Postgres and enqueues slow work into Redis. Python workers run the Audit, Content, and Policy Radar research engines against external APIs through a shared, encrypted key vault. A scheduler drives Policy Radar. Everything runs as Docker Compose services on one VPS behind a TLS proxy.

Section 06 - Module 04 - Mandatory core
Policy Radar (Intelligence)
An always-on engine that keeps every AIOS deployment current with Google. It is platform core, not a per-client feature: it ships in every deployment (Danyal, Haseeb, and beyond) and feeds recommendations into the Command Center. When Google changes or releases something, it auto-researches that change and recommends what to do.

National flagging - every entry tagged
  • Severity - critical / major / minor / info.
  • Category - algorithm / policy / technical / content / local / GEO.
  • Region / nation scope - global vs specific countries, so advice filters to each client's market.
  • Source cited - Search Status Dashboard, Search Central, QRG. Nothing applied on an unverifiable claim.
Command Center

Every recommendation, in one place

Shows all current best practices as a baseline, plus new or changed recommendations triggered by Google moves - what changed, why it matters, regions affected, and the action.

Closed loop

A recommendation can add or update audit checks, adjust content guidance, or raise a client advisory. Human-confirmed before it changes live checks.

Section 09
Infra, Security & Cost Control
Cost control is the reason the platform is cloud-based. The queue, the shared vault, and caching exist to make automated API spend predictable.
Deployment

Docker Compose services: web (Next.js), api (FastAPI), worker (audit + content + research), scheduler (Policy Radar cron + change detect), postgres, redis, proxy (Caddy/Nginx, auto-TLS). Artifacts on the VPS volume; add object storage later if needed.

Recommended VPS

Ubuntu 22.04+, 4 vCPU / 8-16 GB RAM / 100-160 GB NVMe SSD. Hetzner / DigitalOcean / Contabo class. Adan provisions once access is granted.

Cost control - the reason for cloud
  • Central, agency-owned key vault - no per-client keys.
  • Queue concurrency limits + per-run budget caps.
  • Redis caching of expensive API responses.
  • Free / paid tiers gate the costly integrations.
  • Per-job cost estimation + logging, shown to admin.
Security

Secrets

Encrypted key vault + WordPress creds. Keys never logged. Role-based access across the app.

Resilience

Nightly Postgres backups, container restart policies, documented restore, TLS everywhere.

Trust boundary

Audited-site content and generated copy are treated as data, not instructions.

Section 10
Project Scope
A tight v1 protects the 3-5 week timeline. Everything else is parked in a named backlog so new requests never quietly derail the core build.
In scope - v1
  • Audit module (technical + actionable) as a cloud job / API.
  • Content: AIDA drafting, automated schema, AI images.
  • Manual-PDF + WordPress publishing, with review checkpoint.
  • Portal: client, team, admin (manager optional).
  • Policy Radar + Command Center (mandatory core).
  • Agency-provisioned accounts, Google Sheets reporting.
  • Central API key vault + cost controls.
Phase 2
  • Financial audit report (market capacity + revenue).
  • Off-Page module.
  • Fiverr client-data import.
  • Additional CMS connectors (Shopify / Webflow / headless).

Documented now so the next call can turn them into a Phase 2 plan.

Out of scope
  • Self-serve payment gateway (Stripe).
  • Public self-signup.

Scope discipline

New feature requests are logged to the Phase 2 backlog, not slipped into v1 - the guardrail that keeps the timeline honest.
